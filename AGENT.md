# AGENT.md — AI-Driven Development 実証記録

## このドキュメントの目的

本ドキュメントは、本ポートフォリオリポジトリを **AI 駆動開発（AI-Driven Development）** で構築した過程の記録です。

目的は次の二点です。

1. **言語の壁を越え、AI を活用して最速でビジネス要件をプロダクトに落とし込む** 実証実験であること
2. AI を「コードを書かせるツール」ではなく **壁打ちパートナー** として使い、要件定義から実装まで一貫してディレクションできるエンジニアであることを示すこと

採用担当者（リードエンジニア）向けに、**人間が何を判断し、AI が何を実行したか** を透明化しています。

---

## 開発アプローチの概要

```
┌──────────────────────────────────────────────────────────────┐
│  人間（オーナー）                                              │
│  ・ビジネス要件の定義（「本番を汚さないデモモード」）            │
│  ・アーキテクチャ判断（URL 分離 / localStorage / 認証方式）      │
│  ・リスクの特定と指示（5MB 制限、QuotaExceededError）          │
│  ・UI/UX の最終決定・レビュー・品質担保                         │
└────────────────────────┬─────────────────────────────────────┘
                         │ 要件・制約・レビュー
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  AI（Cursor Agent）                                            │
│  ・React / Next.js / TypeScript の実装                         │
│  ・コンポーネント・フック・ユーティリティの記述                  │
│  ・既存コードベースへの統合・リファクタリング                    │
│  ・ビルドエラー・ハイドレーション不一致の修正                    │
└──────────────────────────────────────────────────────────────┘
```

AI は高速な実装力を提供し、人間は **何を作るべきか・何を作ってはいけないか** を定義する責務を担いました。

## 人間と AI の役割分担

### 人間（オーナー）の役割

| 領域 | 具体的な判断・作業 |
|---|---|
| **アーキテクチャ設計** | 本番（`/admin`）とデモ（`/demo/admin`）の URL 分離。サーバー永続化とクライアント永続化の責務分界 |
| **UI/UX 決定** | デモ管理の固定ヘッダー、リセット動線、サイトタイトル編集フォームの配置、重複ナビの削除 |
| **セキュリティ・運用リスク** | 本番 ID/PW の環境変数化、`.env.local` の Git 除外、デモでの認証不要設計 |
| **容量制約への対策** | localStorage 5MB 上限を前提とした画像圧縮仕様（横幅・品質・初期件数）の決定 |
| **品質担保** | ビルド確認、デモと本番の表示一致確認、リセット後のシード件数検証、コミット前の秘密情報チェック |

### AI の役割

| 領域 | 具体的な作業 |
|---|---|
| **コーディング** | App Router ルート、Server Actions、クライアントコンポーネントの実装 |
| **コンポーネント設計** | `gallery-view.tsx` への UI 共通化、`AdminGalleryProvider` / `useGalleryData` の実装 |
| **画像処理** | `compressImageToDataUrl()`（Canvas API）、本番側 `sharp` バリアント生成の維持 |
| **ストレージ層** | `gallery-storage.ts` の read/write/seed/reset、`QuotaExceededError` ハンドリング |
| **認証まわり** | `proxy.ts`（ミドルウェア）、Cookie セッション、環境変数ベースの `admin-config.ts` |
| **デバッグ** | SSR/CSR ハイドレーション不一致（`decodeHtml`）の修正、ビルドエラーの解消 |

## 特筆すべきプロンプト戦略

### ケーススタディ: デモモードの localStorage 設計

単に「管理画面を作って」と依頼するのではなく、**運用上のリスクと技術的制約を先に言語化してから** AI に実装を委譲しました。

#### Step 1 — ビジネス要件の定義（人間）

> 採用担当者がポートフォリオの管理画面を触れる「お試しモード」が欲しい。  
> ただし本番の `works.json` や画像ファイルは絶対に書き換えてはいけない。

この一文から、AI は単純な CRUD ではなく **サンドボックス構成** を提案・実装する方向に進みました。

#### Step 2 — 技術制約の明示（人間）

> データは `localStorage` に保存する。  
> ブラウザの容量上限は約 5MB なので、画像をそのまま Base64 で保存すると `QuotaExceededError` になる。  
> **対策として、アップロード時に Canvas API でリサイズ・圧縮してから保存せよ。**

この指示により、AI は以下を実装しました。

- `lib/gallery-constants.ts` — `IMAGE_MAX_WIDTH` / `IMAGE_JPEG_QUALITY` の定数化
- `lib/gallery-image.ts` — `createImageBitmap` → Canvas 描画 → `toDataURL("image/jpeg")`
- `hooks/use-gallery-data.ts` — `persist()` 内での `QuotaExceededError` 捕捉とユーザー向けエラーメッセージ

#### Step 3 — パラメータのチューニング（人間 → AI）

| 判断 | 経緯 |
|---|---|
| 初期シード件数: 10 → **20** | 面接時に十分な作品数を見せるため。ただし localStorage 容量とのトレードオフを認識 |
| 圧縮横幅: 1000px → **720px** | 20 件 × Base64 でも 5MB 内に収まるよう、面積比（幅の二乗）で容量を試算して決定 |
| 本番との分離確認 | 本番は sharp で 1600px 生成、デモは 720px — 用途ごとに最適化が異なることを明示 |

#### Step 4 — 認証のリスクヘッジ（人間）

> ID/PW をリポジトリに公開しないこと。`.env.local` で簡単に変更できる形にせよ。

AI はハードコードされたフォールバック値を削除し、環境変数未設定時はログイン無効化する実装に変更しました。

## プロンプト設計の原則（本プロジェクトで実践したこと）

1. **制約を先に渡す** — 「何ができないか」（5MB、本番汚染禁止）を機能要件より先に書く
2. **数値で指示する** — 「小さく圧縮」ではなく「横幅 720px、JPEG 70%」
3. **責務の境界を明示する** — 「デモはクライアントのみ」「本番は Server Actions + sharp」
4. **レビュー単位を小さくする** — URL 分離 → localStorage 化 → UI 共通化 → 認証分離、と段階的に進めた
5. **AI の出力を盲信しない** — シード件数が 10 のまま残る問題は、localStorage の既存データが原因と人間が切り分けて確認

## 成果物と学び

### 成果物

- 本番・デモを URL で完全分離した Next.js アプリケーション
- 採用担当者が安全に CRUD を試せるデモモード
- 環境変数ベースの本番認証（秘密情報の非公開化）
- 実装計画書（`gallery-localstorage-implementation-plan.md`）— 要件からタスク分解まで文書化

### 学び

| テーマ | 内容 |
|---|---|
| **AI との協業** | 要件の粒度が実装品質を決める。エッジケースを人間が先に列挙するほど、AI の出力は実用に耐える |
| **制約駆動設計** | localStorage 5MB という制約が、クライアント画像圧縮という合理的な設計判断を生んだ |
| **分離の価値** | デモと本番の URL / データ層 / 画像処理の分離により、それぞれ独立してチューニング可能 |
| **透明性** | AGENT.md のような開発プロセスの可視化自体が、シニアエンジニアへの信頼材料になる |
